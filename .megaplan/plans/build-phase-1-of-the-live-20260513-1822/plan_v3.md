
# Implementation Plan: Live-Update Editor Redesign — Phase 1 Shadow-Mode Vertical Slice (rev 3)

## Overview
Build the v6 redesign (`docs/live-update-editor-redesign.md`) as a shadow-mode dev vertical slice with publishing OFF: **6 new tables — 5 from the spec + a deliberate `topic_editor_runs` provenance table required by the "writes ONLY to new tables" constraint** (deliberate scope expansion over the brief's literal "5-table" wording; flagged here so it's not implicit drift). A native Anthropic tool-use editor (4 read + 6 decision tools), dispatcher with all 4 invariants including the v6 collision-override path, a pure renderer, comparison-trace embeds in dev `#main_test`, and a runner that replays 20+ historical source-message windows. The existing `LiveUpdateEditor`, `LiveUpdateCandidateGenerator`, `ClaudeClient.generate_chat_completion`, and all `live_update_*` tables/accessors stay bit-for-bit untouched. Stop at the Week-1 decision gate.

Env-scoping invariant applies to the **new topic_* tables ONLY**; existing `live_update_*` accessors are not modified, so the env-filter rule has no effect on them. Backfill is a Python script run after migration apply. Dispatcher author resolution uses a **new lean accessor `get_message_author_ids`** (not the heavy `get_live_update_context_for_message_ids`). Tests use a stateful in-memory fake intercepting at the `DatabaseHandler` seam, with a docstring mapping fake-method ↔ production accessor. `pg_trgm` similarity is tested via a **pinned fixture oracle** of expected pairs, not `difflib`. The shadow runner explicitly takes two env knobs (`archive_environment='prod'`, `topic_environment='dev'`) and treats the comparison read of `live_update_*` as telemetry-only. Locked decisions ship as `docs/live-update-editor-phase1-locked-decisions.md` before any code; the defer-audit outcome can append `state='pending_review'` + `request_review_topic` to Steps 2/7 **before** SQL freeze. LOC budget ~800–1000; alarm above 1500.

## Phase 0: Lock decisions (no code)

### Step 1: Defer audit + locked-decisions doc (`docs/live-update-editor-phase1-locked-decisions.md`, `tests/fixtures/historical_headlines.json`, `tests/fixtures/similarity_pairs.json`)
**Scope:** Small
1. **Query** live `live_update_candidates` for the last 30 days counting `decision='defer'` rows by (env, guild); inspect samples for POM action.
2. **Build** two fixtures:
   - `tests/fixtures/historical_headlines.json` — 50 headlines from `live_update_feed_items` + recent `live_update_candidates.raw_output` + recent watchlist titles.
   - `tests/fixtures/similarity_pairs.json` — a **pinned oracle**: ~30 `(headline_a, headline_b, expected_pg_trgm_similarity, expected_match_at_0_55)` rows including both "should collide" and "should NOT collide" pairs drawn from the 50-headline set. Generated by running `SELECT similarity(a, b)` on a real Postgres with `pg_trgm` once; stored as a static fixture. The fake store later asserts against this oracle, not against `difflib`.
3. **Publish** the locked-decisions doc; resolve all 5 locked decisions (canonicalizer, observations retention, collision threshold + override, single-table rejected logging, pending_review).
4. **Defer-audit branch**: if the audit shows ≥5/week acted-upon defers, **append** `state='pending_review'` to Step 2's DDL and a `request_review_topic` tool to Step 7's tool list **before** any DDL is written. Default outcome (under threshold): defer to v2.

## Phase 1: Schema + canonicalizer

### Step 2: Schema migration (`.migrations_staging/20260513XXXXXX_topic_editor_phase1.sql`)
**Scope:** Medium
1. **Author** DDL for the 5 spec tables verbatim from `docs/live-update-editor-redesign.md:100-205`.
2. **Add** the 6th NEW table `topic_editor_runs` (run_id uuid PK, environment, guild_id, live_channel_id NULL, model, started_at, finished_at, tokens_in, tokens_out, latency_ms, status, notes jsonb).
3. **Foreign keys**: declare `topic_transitions.run_id REFERENCES topic_editor_runs(run_id) ON DELETE RESTRICT` so orphan run_ids cannot enter the audit log. Existing FKs from `docs/live-update-editor-redesign.md` are preserved.
4. **Conditional**: if Step 1.4 branch fires, add `state='pending_review'` to the implicit `topics.state` check-list documented in the migration comment (no DB-side enum; values enforced in app code).
5. **Include** `CREATE EXTENSION IF NOT EXISTS pg_trgm;` + `topics_headline_trgm` GIN index + all transitions/sources/topics indexes from spec.
6. **Do NOT** touch any existing `live_update_*` table.
7. **No backfill SQL inside the migration.**

### Step 3: Canonicalizer + fuzz test (`src/features/summarising/topic_canonicalizer.py`, `tests/test_topic_canonicalizer.py`, `tests/test_topic_similarity_oracle.py`)
**Scope:** Small
1. **Implement** `canonicalize(proposed_key, headline, source_messages=None) -> str` (pure, no DB).
2. **Implement** `similarity_score(headline_a, headline_b, authors_a, authors_b) -> dict`. Production uses `pg_trgm` via the new DB accessor; this Python helper computes author_overlap + accepts a `trigram_similarity` value passed in.
3. **Test** canonicalizer with `historical_headlines.json`.
4. **Pinned-oracle test** (`test_topic_similarity_oracle.py`): for each row in `similarity_pairs.json`, assert the dispatcher's match/no-match decision at threshold 0.55 equals the oracle's `expected_match_at_0_55`. This is what gates the threshold tuning — `difflib` is used inside the fake store only as a placeholder retrieval signal, never as the oracle.

### Step 4: Python backfill script (`scripts/backfill_topics_from_feed_items.py`)
**Scope:** Small
1. **One-shot, dev only.** Pulls `live_update_feed_items` filtered to `status='posted' AND len(discord_message_ids) > 0` (skips pending/failed legacy rows so `publication_status='sent'` stays truthful).
2. **For each row**: parse JSONB `source_message_ids`, resolve authors via the new lean `get_message_author_ids(message_ids)` accessor (NOT the heavy context accessor), call `topic_canonicalizer.canonicalize`, insert `topics` (`state='posted'`, `publication_status='sent'`, `discord_message_ids` populated), then `topic_sources` rows (one per `(topic_id, bigint(message_id))`). Idempotent on `(environment, guild_id, canonical_key)`.

## Phase 2: Tool-use editor + dispatcher

### Step 5: Native tool-use client (`src/common/llm/topic_editor_client.py`)
**Scope:** Medium
1. **Add** a sibling client to `ClaudeClient` (do NOT modify `claude_client.py:33-92`). Wrap `messages.create` with `tools=`, `tool_choice='auto'`, multi-turn tool-result loops, extended-thinking blocks, surfacing `tool_use_id`.
2. **Expose** `run_tool_loop(system_prompt, user_msg, tools, dispatcher, max_turns=12) -> RunTrace`. Transport-only.

### Step 6: Storage + DB accessors for new tables (`src/common/db_handler.py`, `src/common/storage_handler.py`)
**Scope:** Medium
1. **Add** in `db_handler.py` (async-in-thread), each backed by a matching `storage_handler.py` payload-builder (DEBT-028 two-layer pattern):
   - `create_topic`, `update_topic_state`, `get_topic_by_id`, `get_topic_by_canonical_key`
   - `add_topic_sources` (handles `(topic_id, message_id)` UNIQUE)
   - `add_topic_alias`, `resolve_topic_by_alias`
   - `record_topic_transition` (handles `topic_id=NULL` for rejected creates; validates `run_id` exists in `topic_editor_runs` to surface FK violations early)
   - `record_editorial_observation`
   - `search_topics_by_headline(headline, environment, guild_id, ...)` — trigram + alias join, returns rows with their trigram score so the dispatcher can apply the threshold without a second round-trip
   - `list_watching_topics(environment, guild_id)`
   - `create_topic_editor_run`, `finish_topic_editor_run`
   - `get_topic_transition_rates(environment, guild_id, window_hours=168)` — rolling aggregate by action
   - **NEW**: `get_message_author_ids(message_ids: list[int]) -> dict[int, int]` — lean accessor returning `{message_id: author_id}` from `discord_messages`. Replaces the heavy `get_live_update_context_for_message_ids` on the dispatcher hot path AND for the backfill script. Existing accessor is untouched and still used by prompt assembly.
2. **Mirror** every method as a `storage_handler.py` counterpart with the hardcoded payload keys.
3. **Env-scoped read invariant (applies to NEW topic_* tables ONLY)**: every new read accessor filters by `(environment, guild_id)` mandatorily; `cross_env: bool = False` is the only escape hatch. **Existing `live_update_*` accessors are NOT modified** — the env-filter rule does not retroactively apply to them.
4. **Do NOT** modify any existing `live_update_*` accessor.

### Step 7: 10 tool schemas + dispatcher (`src/features/summarising/topic_tools.py`, `src/features/summarising/topic_dispatcher.py`)
**Scope:** Large
1. **Define** the 10 Anthropic tool schemas verbatim from `docs/live-update-editor-redesign.md:264-391`. Write tools accept `override_collisions: list[{topic_id, reason}]`. `search_topics` is dispatcher-scoped to `(environment, guild_id)`. If Step 1.4 branch fired, add `request_review_topic` as an 11th tool.
2. **Implement** `TopicDispatcher.dispatch(tool_name, tool_use_id, args, run_ctx)`. Author resolution is authoritative via `_resolve_authors(message_ids)` → **`get_message_author_ids`** (lean accessor, NOT the heavy context call).
   - **Inv 1 (structural)**: `post_simple_topic` returns tool-error when authoritative `distinct_authors >= 2` OR `len(source_message_ids) >= 3`; writes `action='rejected_post_simple'`, `topic_id=NULL`.
   - **Inv 2 (collision + override)**: canonicalize → call `search_topics_by_headline` (env-scoped, returns rows + trigram score); compute `author_overlap` against authoritative archive authors. Match (canonical-key-prefix OR trigram ≥ 0.55 AND author_overlap ≥ 1) AND `topic_id` not in `override_collisions` → tool-error + log `action='rejected_*'`. Override present → accept the write + log `action='override'` per matched topic_id with `reason`.
   - **Inv 3 (idempotency)**: `(run_id, tool_call_id)` check first.
   - **Inv 4 (source uniqueness)**: rely on `topic_sources` UNIQUE; on conflict log `action='rejected_*'`.
3. **Every** rejection writes a `topic_transitions` row with `topic_id=NULL` for failed creates.
4. **Tests** (`tests/test_topic_dispatcher_invariants.py`, `tests/test_topic_aliases_resolution.py`) use the fake store from Step 11.

### Step 8: Editor agent + prompt (`src/features/summarising/topic_editor.py`, `src/features/summarising/topic_editor_prompt.py`)
**Scope:** Medium
1. **Build** `TopicEditor.run_once(window, environment, guild_id)`:
   - Insert a `topic_editor_runs` row via `create_topic_editor_run` (NEVER `live_update_editor_runs`).
   - Assemble context: new source messages, watching topics (`list_watching_topics`, env-scoped), recently-posted topics.
   - Call `topic_editor_client.run_tool_loop` with `TopicDispatcher`.
   - Close via `finish_topic_editor_run`.
   - Return a `RunTrace`.
2. **Author** the editor prompt covering taste from `_meets_editorial_bar`; cap `record_observation` at ~3/run; mention `request_review_topic` only if Step 1.4 fired.
3. **Note**: concurrent-run lease deferred to Phase 1.5; trace embed surfaces `lease: DEFERRED`.

## Phase 4: Test fakes + replay runner + tests

### Step 11: Stateful fake store + caller map (`tests/conftest.py`, `tests/fakes/topic_store.py`)
**Scope:** Medium *(rescoped from Small — this is a stateful shim, not a stateless conftest stub)*
1. **Build** `tests/fakes/topic_store.py` as a **stateful** in-memory fake intercepting at the **`DatabaseHandler` seam** (NOT the `StorageHandler` lower layer) — production parity for callers. Backed by Python dicts/sets with explicit modeling of: env-scoped indexes by `(environment, guild_id)`, `(topic_id, message_id)` UNIQUE enforcement, `(environment, guild_id, alias_key)` UNIQUE alias resolution, idempotency on `(run_id, tool_call_id)`, per-run transition log, and `run_id` FK validation against fake `topic_editor_runs`.
2. **Caller map (mandatory module docstring)**: a one-table mapping listing every fake method ↔ the `DatabaseHandler` accessor it satisfies, including exact signature and return shape. The fake fails type-check / signature-introspection if it drifts.
3. **Similarity in the fake**: the fake uses the `similarity_pairs.json` oracle for any pair present in the oracle (returns the pinned score); for unknown pairs it uses `difflib.SequenceMatcher.ratio` ONLY as a retrieval placeholder — threshold-tuning tests must use oracle-covered pairs.
4. **Wire** the fake into dispatcher / alias / shadow tests via dependency injection on `TopicDispatcher` and `TopicEditor`.

### Step 12: Shadow-mode replay runner (`scripts/run_live_update_shadow.py`)
**Scope:** Medium
1. **Model on** `scripts/run_live_update_dev.py`; do NOT route through `LiveUpdateEditor`.
2. **Two env knobs (mandatory)**: `--archive-env prod` (read source messages, `live_update_candidates`, `live_update_feed_items`) and `--topic-env dev` (write `topic_*` rows + `topic_editor_runs`). Default both for the Week-1 run; surface them in the trace embed header to prevent cross-wiring.
3. **Comparison read is telemetry-only**: explicitly document that the `live_update_candidates` / `live_update_feed_items` read populates the divergence field of the trace embed and never feeds back into dispatcher state — satisfies the brief's "do not read for decisions" constraint.
4. **Load** ≥20 historical source-message windows; flags `--windows N`, `--lookback HOURS`, `--once`.
5. **Per window**: run editor with publishing OFF; post one trace embed per window to `DEV_SHADOW_TRACE_CHANNEL_ID`.
6. **Aggregate** summary embed at end.
7. **Deliberately** roll one window twice; trace embed asserts alias resolution to the same `topic_id`.

### Step 13: End-to-end shadow smoke test (`tests/test_shadow_mode_replay.py`)
**Scope:** Small
1. **Smoke test** using the fake store: fixture window → new editor publishing OFF → assert `topic_transitions` rows recorded, no `channel.send` invoked (mocked), trace-embed payload contains all required fields including override rate, observation rate, archive-env / topic-env header, and `lease: DEFERRED`.
